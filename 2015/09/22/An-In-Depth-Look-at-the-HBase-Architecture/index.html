[译] HBase 架构总览

一直想写一篇关于 Apache HBase 架构的文章，后来看到 MapR 的一篇博客中已经有了一篇，文章的组织和结构都比较清楚，于是拿来翻译一下帮助自己理解，原文中比较啰嗦的地方我在翻译过程中精简了，排版也稍微修改了一些。

HBase 架构组成

HBase 的架构属于主从类型，由三种 Server 构成：

  • Region Server：负责数据的读写，客户端在操作数据时，直接与 HBase 的 Region Server 们进行通信。
  • Master：Region 的分配、DDL（增删表）操作都在 HBase 的 Master 上完成。
  • Zookeeper：维持集群的运行状态。

Region Server 上的数据都存放在 Hadoop 的 DataNode 上的，HBase 上的所有数据都存放在 HDFS 上。HBase 的 Region Server 本身也是 Hadoop 的 DataNode，这样就保证了 Region Server 的数据本地化（把数据放得更靠近最需要用数据的地方）。HBase 的数据在刚写入的时候是存放在本地的，但是如果中间一个 region 发生了移动，那么这份数据就不在本地了，直到发生一次 HBase 的 Major Compaction。

Regions

HBase 的表根据 row key 的范围分成了很多个“Regions”，每个 region 包含了该 region 起始 row key 和 终止 row key 之间包含的所有行的数据，这些 region 会被分配到集群各个 region server 对应的节点上，并提供数据的读写服务，一个 region server 可以管理约 1000 个 region。

Master

HBase Master 主要负责以下事项：

  • 协调各个 region server
    • 启动的时候分配 region，在恢复或负载均衡的时候重分配 region
    • 监控集群上所有 region server 实例的生存状态（通过监听 ZooKeeper 的通知）

  • 管理功能

    • 提供增删改表的接口

ZooKeeper：协调大师

HBase 使用 ZooKeeper 的分布式协调服务来维护集群中 server 的状态。ZooKeeper 维护着每个活着可用的 server，并提供 server 的故障通知。Zookeeper 使用一致性算法来保证共享状态的一致性，所以 ZooKeeper 一般需要 3 或 5 台机器。

各个成员的相互合作

ZooKeeper 用于协调分布式系统中各成员间共享的状态信息，HBase Region Server 们和活着的 HBase Master 连接到 ZooKeeper 之后各自创建一个 ephemeral 节点，ZooKeeper 通过心跳机制来维护这些节点的活跃状态。

每一个 region server 都会创建一个 ephemeral 节点，HMaster 通过监控这些节点来发现可用的 region server 和监控 region server 的故障。HMaster 们通过竞争创建一个 ephemeral 节点，ZooKeeper 会选举出一个节点并保证只有一个 master 是生效的。生效的 HMaster 会向 ZooKeeper 发送心跳，而不活跃的 HMaster 会监听生效的 HMaster 的故障通知。

如果一个 region server 或者活跃的 HMaster 发送心跳失败，他们与 ZooKeeper 之间的连接就会中断，对应的 ephemeral 节点也会被删除，这些节点的监听者就会被通知。活跃的 HMaster 监听 region server 们的状态，并在故障的时候恢复 region server；非活跃的 HMaster 监听活跃 HMaster 的状态，并在活跃 HMaster 故障的时候自动变成活跃的 HMaster。

HBase 的第一次读写

HBase 中有一张特殊的目录表：META 表，这张表保存了集群所有 region 的位置信息，ZooKeeper 保存了 META 表的位置信息。

客户端向 HBase 发送第一次读或者写请求时的步骤是这样的：

  • 客户端从 ZooKeeper 获取到存储 META 表的 region server。
  • 客户端获取想访问的 row key 所在的 region server。
  • 客户端从对应的 region server 上获取访问的行。

客户端会缓存 META 表和之前读过的 row key 的信息，在之后的读写中，这个客户端就无需再查询 META 表，除非有 region 迁移的情况导致查询失败，这样的情况下客户端会重新查询该 row key 的信息并更新本地的缓存。

HBase META 表

  • META 表是 HBase 的一张表，保存了系统中所有 region 的信息。
  • META 实际上就是一个 B Tree。
  • META 表的结构如下：
    • Key：region 的起始 row key，region ID
    • Values：region server

Region Server 组件

Region Server 在 HDFS data node 上运行，包含以下组件：

  • WAL (Write Ahead Log)：预写式日志是分布式文件系统中的文件，存放还没被持久化到永久存储的数据，可在故障后恢复数据时使用。
  • BlockCache：读缓存，在内存中保存最近读过的数据，使用的是 LRU 算法。
  • MemStore：写缓存，保存还未被写入磁盘的新数据，数据在写入之前会先排序，每个 region 的每个列族对应一个 MemStore。
  • HFile：在磁盘中以 Key Value 的形式存放排序过后的行。

HBase 写数据步骤 I：WAL

当客户端发起一次写请求的时候，第一步是将数据写入 WAL：

  • 数据的操作会被添加到 WAL 文件（保存在硬盘上）的末尾。
  • WAL 用于服务器故障后未持久化数据的恢复。

HBase 写数据步骤 II：MemStore

数据被写入 WAL之后，会接着保存在 MemStore 中，然后客户端将会收到数据已被接受的响应。

MemStore 将写入的数据按照排序好的 Key Value 的形式存放于内存中，数据的排列顺序跟在 HFile 中的一致。每一个列族都会对应一个独立的 MemStore。

HBase 写数据步骤 III：HFile

当 MemStore 中积累了足够多的数据，所有排序后的数据就会写入到 HDFS 上的一个新的 HFile 中。每个 HBase 的列族会对应一个或多个 HFile，HFile 中存放了 Key Value 的实例，并随着数据的不断写入不断被创建。

正如刚才说过的，每个列族都会对应一个独立的 MemStore，如果一个 MemStore 的大小达到了上限，所有的 MemStore 都会将数据写入磁盘，即磁盘的写入是 region 级别的，非 MemStore / 列族 级别，这也是一个 HBase 表不能有太多列族的原因。

每次写入磁盘之后，最大的写入序列号会被保存在每个 HFile 的 meta 字段，用来表示上次持久化的位置。当一个 region 启动的时候，序列号们就会被读取，然后最大的一个用来标识新数据开始的位置。

由于 HFile 的写入是按顺序写，减少了磁盘指针的移动，所以效率比较高。

HFile 的结构

HFile 中包含一个多级的索引，这使得 HBase 不用读取整个文件就可以找到想要的数据，这个多级的索引就像一个 B+Tree：

  • Key Value 对按照升序保存
  • Key Value 的数据会分块，每块 64 KB
  • 每个数据块都有自己的 leaf index
  • 每个数据块的最后一个 row key 会存放在 intermediate index 中
  • root index 指向 intermediate index

在 HFile 的末尾，存在一个指向 meta 块的 trailer，trailer 同时还包含其它的信息：

  • Bloom Filter：通过 row key 过滤，减少需要查询的文件
  • Time Range：通过时间过滤，减少需要查询的文件

HFile 的索引会在该文件被打开的时候装载入内存，提高读的效率。

HBase 读数据步骤

从上面 Region Server 的组件可以看到，一个 row key 对应的数据可能存在于多个地方：持久化的数据在 HFile 中，最近更新的数据在 MemStore 中，最近读过的数据在 Block Cache 中。当客户端发来一个读请求的时候，HBase 读数据的步骤如下：

  • 首先读采用 LRU 算法的 Block Cache，这里保存了最近读过的数据。
  • 接着读 MemStore，这里保存了最近更新过的数据。
  • 如果前两步没有找到数据，HBase 会接着根据 HFile 的索引和 Bloom Filter 将目标 HFile 载入内存，然后获取目标 row key 对应的数据。

正如之前说的，每个 MemStore 可能会对应多个 HFile，这意味着每一次读请求会将多个 HFile 都检查一遍，这种读放大的现象降低了读的效率。

HBase Compaction

  • Minor Compaction：通过合并排序，自动将多数个小 HFile 合并成少数个大的 HFile。

  • Major Compaction：将同一个 region 同一个列族的所有 HFile 合并为一个 HFile，并在此过程中删除已经过期或者带有删除标记数据，将非本地的 region 数据移动到 region server 本地。这会提高读数据的效率，但是由于会重写所有的文件，这种写放大会造成大量的磁盘和网络 IO。

Region：连续的键

HBase Region 概览：

  • 一张表可以水平地被分成一个或多个 region，一个 region 包含起始键和终止键之间排序过的列。
  • 每个 region 默认大小是 1 GB。
  • 客户端直接通过 region server 访问 region。
  • 一个 region server 上可以有大概 1000 个 region（可以属于不同的表）。

起始的时候（译注：在没有做手动 split 的前提下）一张表只有一个 region，当此 region 增长到一定大小的时候，就会被分成两个 region，这两个 region 还是在原来的 region server 上，并且这次分裂会汇报给 HMaster。

为了负载均衡，HMaster 可能会将新的 region 移动到其它 region server 上去，这样就导致了 region 数据非本地化（数据存放在另外一台 HDFS data node 上）。不过再经过一次 major compaction 之后，数据又会变成本地化。

故障后的数据恢复

发生故障之后，HBase 会重新读 WAL 文件，将数据的更新操作重新写入 MemStore 并最终持久化到 HFile中；如果一个 region server 挂了，这个时候 ZooKeeper 会通过心跳机制监测到这种故障并报告给 HMaster，HMaster 会将这台 region server 上的所有 region 分配给其它活着的 region server 们，这台 region server 上的 WAL 文件也会被分成多个独立的文件分发到其它活着的 region server 上去，然后其它 region server 会将这些 WAL 中包含的数据更新重新写入 MemStore。
