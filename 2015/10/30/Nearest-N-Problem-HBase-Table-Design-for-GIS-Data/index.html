附近的 N 个 XX 问题：HBase GIS 数据表设计

在《HBase in Action》这本书中，其中一章“Scaling GIS on HBase”，讲得是针对 GIS 数据应用场景的数据表设计和使用，我觉得内容比较有意思，所以在此记录。

1. 关于附近的 N 个 XX 问题和 GIS 数据

这是我们生活中常常遇到的问题，比如我们到了一个陌生的地方，打开智能手机中的 APP，就能搜索到附近离我们最近的餐馆，这个过程发生了两个重要的数据操作：

  • 位置定位：APP 会首先采集我们所在的地理位置数据，即经纬度等信息。
  • 数据查询：采集完成之后服务器会进行一次数据查询，在所有的有地理位置记录的餐馆中筛选得到离我们最近的 N 个餐馆。

这是一个典型的附近的 N 个 XX 的问题，位置定位和查询中的主要数据便是 GIS（Geographic Information System）数据，可以简单地理解为经度、纬度、海拔等数据等数据的综合。而且随着智能设备的普及，这类的数据采集越来越方便，量也越来越大，能有效地解决这类问题还是比较有意义的。

2. HBase 数据表的设计

说到 HBase 的表设计，最多的就是 row key 的设计，合适的 row key 设计能够让我们用最少的扫描获取到所有感兴趣的数据。为了达到这个目的，需要满足两个要求

  • 地址位置接近的数据点，在磁盘上存储的位置也应该尽可能接近。
  • 一次查询，扫描的点越少越好。

2.1 最简单粗暴的方案

拿到这样一个需求之后，一个非常粗暴的方案就是将 row key 设计成经度和纬度的简单拼接，这样经度接近的数据点在磁盘中就存放地越近，查询

但是稍微思考一下就知道这种方案的弊端，如果两个数据点的经度接近，纬度差很多的话，其实也会被存放在相邻的位置，这样我们进行数据查询的时候，就会多出来很多无用的点（离我们目标点的距离太远）。

2.2 降维：geohash

简单的拼接不能很好地解决问题，归根到底主要是因为 GIS 数据有至少两个维度：经度和纬度，两个维度之间是独立的，也没有优先级的关系，而 row key 是一维的，所以如果要在 row key 中包含经纬度的信息，必须对数据进行降维处理，而 geohash 算法正好能满足这个需求。

下图是 geohash 算法的简介，简单讲 geohash 就是对地球表面进行切块，例如：纬度的范围是 [-90, 90]，第一刀切成 [-90, 0] 和 [0, 90] 两块，如果输入点落在了上半区，第一位记为 1，依此类推，经度也是用同样的方法。

这样根据最初输入的经纬度，在一定的精度条件（切的刀数）下，就能计算出每个点的 geohash 值，这个值可以作为 HBase 数据表的 row key，因为 geohash 值前缀相同的位数越多，两个点在地理位置上越接近。

这样我们在进行数据查询的时候，步骤应该是这样的：

  • 计算输入数据点的 geohash 值。
  • 根据输入的 geohash 值的前缀，在 HBase 中使用 row key filter 筛选得到可能的结果数据点。
  • 对所有可能的结果数据点，计算与目标点的距离，排序后输出最近的 N 个。

3. geohash 的问题

3.1 邻近两点的 geohash 值可能相差很多

在这个问题上，geohash 并不是万能的。因为从本质上看，geohash 是把地球切成了一个个的小块，每一个小块内的点拥有相同的 geohash 值，但是相邻小块的 geohash 值有可能差得比较多。举个极端点的例子：现在有两个点，分别是（90.000000, 0.000001）和（90.000000, -0.000001），这两点从经纬度上看位置十分接近，但是两点的 geohash 值分别是 upbpbpbpbpbx 和 gzzzzzzzzzzr。

解决办法是将感兴趣小块周围的八个小块一起纳入扫描的范围，如下图所示，这样能比较有效地避免 geohash 的边界问题。

3.2 扫描的精度该设置多少

在前面所述的条件下，我们数据查询的时候实际上是对 HBase 的一张表进行 row key prefix filter，但是此 filter 使用的前缀的位数如何确定？位数越大，精度越高，但可能得到的数据点会过少；反之精度降低，导致得到的数据点过多，造成了过多的 IO 等资源的消耗。

实际上这个值要根据实际的数据进行测试决定，可以通过实验先取一个经验值，或者根据用户的其它输入条件确定（比如距离在多少公里之内）。

4. 写在最后

其实书中此章还介绍了另外一种 GIS 数据的查询场景：查询落在某个多边形范围内的数据点，不过换汤不换药，表的设计是不变的，查询思路也类似。从这样的案例中可以看到，在 HBase 数据表的设计中：

  • 尽量把查询需要的纬度数据放在 row key 中，row key 即索引呀。
  • 不要期待单纯的 row key 的设计能够解决所有问题，很多时候跟其它已有的方法结合才能发挥出 HBase 最强大的功力。
  • 题外话：如果上边两个都解决不了问题，试试 HBase 的二级索引，或者试试 HBase 之外的解决方案 :)
